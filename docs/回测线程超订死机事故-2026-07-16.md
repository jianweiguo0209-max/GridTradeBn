# 回测线程超订死机事故 — 2026-07-16

## 一句话

本地跑多进程回测时 `workers=8`，每个 worker 进程的 numpy(OpenBLAS) 又各自开满 8 核线程 →
**8×8 = 最多 64 条线程抢 8 个核** → load average 飙到 29 → 调度 thrash 饿死 `watchdogd`
→ 内核 **watchdog-timeout panic 强制重启**（当天两次）+ 一次过载假死。**根因是 CPU 线程超订，不是内存耗尽。**

## 影响

- 2026-07-16 当天 **两次内核 panic 重启**（12:38、13:45）+ 一次过载假死（约 14:2x，未重启但进程被杀）。
- 每次重启清空 `/private/tmp` scratchpad，回测脚本 + 结果丢失，多轮重跑。
- 币安减宽度(band)扫描被反复腰斩。

## 时间线与硬证据

| 时间 | 事件 | 证据 |
|------|------|------|
| 12:38 | 内核 panic 重启① | `/Library/Logs/DiagnosticReports/panic-full-2026-07-16-123849.0002.panic`，panicString = `watchdog timeout: no checkins from watchdogd in 94 seconds`，崩溃时 compressor 压着 ~8GB(71% 上限) |
| 13:45 | 内核 panic 重启② | `panic-full-2026-07-16-134535.0002.panic`，`watchdog timeout ... 92 seconds`，compressor ~8GB(68%)；"重启监控后仅 349 次心跳"→ 重启后跑上重活很快又卡死 |
| ~14:2x | 过载假死（非重启） | `uptime` 开机 55 分钟无新 panic；`load average 7.75 / 29.26 / 27.09`（8 核机器 load 29）；band_scan 进程被杀（日志尾 `resource_tracker: leaked semaphore at shutdown` = 异常终止） |

## 根因

核实四条，全部指向 CPU 线程超订：

| 检查 | 结果 | 含义 |
|------|------|------|
| `OPENBLAS_NUM_THREADS` 等线程环境变量 | **全未设** | 每进程开满核 |
| `numpy.show_config()` BLAS 后端 | **OpenBLAS**（非 Accelerate） | 每进程默认开 =核数(8) 条线程 |
| `gridtrade/backtest/*.py` 线程限制 | **无**（grep 确认） | 没人拦 |
| 崩溃/假死时 load average | **29**（8 核） | 严重超订铁证 |

- **不是主动重启**：两次都是内核 panic，且精确对上 `last reboot` 的 12:38/13:38。
- **不是用户态 OOM**：若是 OOM，macOS jetsam 会杀 python、机器不重启；日志**无 jetsam 杀进程记录**。
- **内存是帮凶不是主因**：前两次 compressor 确扛着 8GB，但 0.5GB 的单窗负载都能把机器搞假死——只有 load=29 能解释，内存解释不了。贯穿三次的主因是**线程超订**。

## 诊断方法（下次照抄）

1. **是否真重启**：`uptime`（开机时长）+ `last reboot` + `ls -lt /Library/Logs/DiagnosticReports/*.panic`（panic 报告穿越重启存活）。
2. **panic 定性**：panic 文件头是 JSON，取 `panicString`。`watchdog timeout` = 系统卡死（泛化），需再看内存/CPU 佐证。
3. **内存 vs CPU**：panic 里找 `Compressor Info`（压了多少）；`load average` 对比核心数（load ≫ 核心 = CPU 超订）。
4. **反例证伪**：小负载(0.5GB)也崩 → 排除纯内存假设。别只查一个维度就下结论。

## 修复（已焊入代码，防复发）

- **`gridtrade/backtest/_resource_guard.py`**（新）：`apply_thread_caps()` 把 5 个 BLAS/OpenMP 线程变量 setdefault=1；`safe_workers()` 把 worker 数夹到 ≤ 半数核心并打印告警。
- **`gridtrade/backtest/__init__.py`**：导入即调 `apply_thread_caps()`——先于所有子模块、先于 numpy，worker 子进程经 `os.environ` 继承 1 线程。本包不被实盘 import，不碰实盘。
- **`gridtrade/backtest/backtest_run.py`**：顶部 `import pandas` 前再锁一道（此入口 pandas 早于 backtest 导入）。
- **`scripts/sweep_run.py`**：`workers = safe_workers(BT_WORKERS)`，超半数核心自动夹并告警。

**验证**：清空环境变量起干净 python，`import gridtrade.backtest` 后 `OPENBLAS_NUM_THREADS=1`（且 numpy 尚未载入）；`safe_workers(8)→4`；线程锁后 band 扫描 load 稳在 ~5（对比事故时 29）。

## 长跑额外防护（人工）

- 单窗独立进程串行（避免多窗 1m 数据挤一进程的内存累积 → `preload_window` 静默丢格）。
- 挂 load 熔断监视器：1min load 连续 >14 就 `pkill` 扫描，物理杜绝再假死。

## 教训

1. **别只查一个维度**：第一版结论"是内存"漏了 CPU 线程；小负载也崩的反例才逼出真因。
2. **多进程 + numpy 必锁 BLAS 线程**：`workers` 是进程数，每进程的 BLAS 会再乘核数，不锁就是 N×核数 的线程爆炸。这是标准坑，不是本项目特有。
3. **护栏焊进代码，别靠记得 export**：环境变量靠人记必忘；放包 `__init__` 才是"想犯都难"。
