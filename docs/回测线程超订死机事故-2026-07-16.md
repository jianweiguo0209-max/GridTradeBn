# 回测线程超订死机事故 — 2026-07-16

## 一句话

本地跑多进程回测反复死机（两次内核 watchdog panic + 一次假死）。**两个叠加的坑，主次一度搞反，以此更正为准：**

1. **主因**——临时 scratchpad 脚本（band_scan/trail_ab）缺 `if __name__ == '__main__':` 守卫。macOS 默认 **spawn** 启动，`ProcessPoolExecutor`（`simulate_tasks` 在 `workers>1 且 payload>1` 时用）的子进程会**重执行整个脚本**；无守卫 → 子进程重跑 `preload_window`（重载整窗 1m 数据）+ 再 spawn → 递归资源爆炸（四窗驱动每子进程各重载 4 窗）→ 假死/panic，最终 `RuntimeError`。**仓库正式入口（sweep_run.py:88、backtest_run.py:635）都有守卫，坑只在临时脚本。**
2. **帮凶**——numpy(OpenBLAS) 每进程默认开 =核数 条线程，`workers×核数` 理论超订。但网格仿真不吃 BLAS matmul、线程多半空转，单独未必顶到 load 29，是①递归重载叠加时的放大器。

> ⚠️ 诊断三跳（内存 → BLAS 线程 → 守卫）才定案。本文最初版把"CPU 线程超订"当唯一根因**不完整**，直到多进程脚本抛出自带诊断的 `RuntimeError` 才看清主因是①守卫，据此更正。

## 影响

- 2026-07-16 当天 **两次内核 panic 重启**（12:38、13:45）+ 一次过载假死（约 14:2x，未重启但进程被杀）。
- 每次重启清空 `/private/tmp` scratchpad，回测脚本 + 结果丢失，多轮重跑。
- 币安减宽度(band)扫描被反复腰斩。

| 13:45 | 内核 panic 重启② | `panic-full-2026-07-16-134535.0002.panic`，`watchdog timeout ... 92 seconds`，compressor ~8GB(68%)；"重启监控后仅 349 次心跳"→ 重启后跑上重活很快又卡死 |

## 根因

核实四条，全部指向 CPU 线程超订：

| 检查 | 结果 | 含义 |
|------|------|------|
| `OPENBLAS_NUM_THREADS` 等线程环境变量 | **全未设** | 每进程开满核 |
| `numpy.show_config()` BLAS 后端 | **OpenBLAS**（非 Accelerate） | 每进程默认开 =核数(8) 条线程 |
| `gridtrade/backtest/*.py` 线程限制 | **无**（grep 确认） | 没人拦 |
| 崩溃/假死时 load average | **29**（8 核） | 严重超订铁证 |

- **不是主动重启**：两次都是内核 panic，且精确对上 `last reboot` 的 12:38/13:38。
- **不是用户态 OOM**：若是 OOM，macOS jetsam 会杀 python、机器不重启；日志**无 jetsam 杀进程记录**。
- **内存不是主因**：前两次 compressor 确扛着 8GB，但 0.5GB 的单窗负载都能把机器搞假死——内存解释不了。
- **主因是缺 `__main__` 守卫的递归重执行**：`RuntimeError` 铁证 + traceback 显示子进程重跑脚本到 `run_arm` 行再 spawn。它同时解释了 load 29（子进程递归重载整窗数据）；网格仿真本身不吃 BLAS matmul，光靠线程超订顶不到 29。**线程超订是叠加放大器，不是主因。**（表中 load=29 是"机器确被超订"的铁证，但超订的源头是递归重执行，非 BLAS。）

## 诊断方法（下次照抄）

1. **是否真重启**：`uptime`（开机时长）+ `last reboot` + `ls -lt /Library/Logs/DiagnosticReports/*.panic`（panic 报告穿越重启存活）。
2. **panic 定性**：panic 文件头是 JSON，取 `panicString`。`watchdog timeout` = 系统卡死（泛化），需再看内存/CPU 佐证。
3. **内存 vs CPU**：panic 里找 `Compressor Info`（压了多少）；`load average` 对比核心数（load ≫ 核心 = CPU 超订）。
4. **反例证伪**：小负载(0.5GB)也崩 → 排除纯内存假设。别只查一个维度就下结论。
5. **等确定性错误再定主因**：`watchdog timeout` 是泛化卡死，只能推测；直到多进程脚本抛 `RuntimeError: ... start a new process before ... bootstrapping`（自带诊断，直指缺 `__main__` 守卫）才敢定。间接证据（compressor/load）只能缩小范围，别据此拍板"根因"。

## 修复

**① 主因修复 — 临时脚本加 `if __name__ == '__main__': main()`**（执行代码放 main()，顶层只留 import/def）。band_scan.py 补守卫 + 调 import 顺序后：扫描正常产出、不再 RuntimeError、load 稳 ~4。仓库正式入口本就有守卫，无需改。此坑靠人写脚本时记得 → 已入项目记忆提醒。

**② 帮凶修复 — 锁 BLAS 线程 + 夹 worker（已焊入代码，防复发）：**

- **`gridtrade/backtest/_resource_guard.py`**（新）：`apply_thread_caps()` 把 5 个 BLAS/OpenMP 线程变量 setdefault=1；`safe_workers()` 把 worker 数夹到 ≤ 半数核心并打印告警。
- **`gridtrade/backtest/__init__.py`**：导入即调 `apply_thread_caps()`——先于所有子模块、先于 numpy，worker 子进程经 `os.environ` 继承 1 线程。本包不被实盘 import，不碰实盘。
- **`gridtrade/backtest/backtest_run.py`**：顶部 `import pandas` 前再锁一道（此入口 pandas 早于 backtest 导入）。
- **`scripts/sweep_run.py`**：`workers = safe_workers(BT_WORKERS)`，超半数核心自动夹并告警。

**验证**：清空环境变量起干净 python，`import gridtrade.backtest` 后 `OPENBLAS_NUM_THREADS=1`（且 numpy 尚未载入）；`safe_workers(8)→4`；线程锁后 band 扫描 load 稳在 ~5（对比事故时 29）。

## 长跑额外防护（人工）

- 单窗独立进程串行（避免多窗 1m 数据挤一进程的内存累积 → `preload_window` 静默丢格）。
- 挂 load 熔断监视器：1min load 连续 >14 就 `pkill` 扫描，物理杜绝再假死。

## 教训

1. **拿到确定性错误再定根因**：诊断三跳（内存→BLAS→守卫）里前两跳都拿间接证据（compressor 8GB、load 29）猜维度、都偏了；`RuntimeError` 自带诊断才让主因水落石出。间接证据只缩小范围，别据此拍板。
2. **多进程脚本必须 `if __name__ == '__main__':`**：macOS/Windows 默认 spawn，子进程重执行主模块；无守卫 → 递归重跑 + 资源爆炸 + RuntimeError。这是本次死机真凶，标准坑非本项目特有。
3. **多进程 + numpy 顺手锁 BLAS 线程**：`workers` 是进程数，每进程 BLAS 再乘核数；虽非本次主因，BLAS-重 的负载下不锁会真超订。
4. **护栏焊进代码，别靠人记**：线程锁已入包 `__init__`（想犯都难）；`__main__` 守卫只能靠写临时脚本时记得 → 已入项目记忆提醒。
