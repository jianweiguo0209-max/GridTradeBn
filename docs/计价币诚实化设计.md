# 计价/结算币诚实化设计（config / `quote_currency` 驱动 canonical 符号）

> 状态：**P1 代码+测试已完成**（2026-06-30），待 P2 testnet 切换（A 等 flat）。§11 三个开放问题已定（config 本期加 / 切换走 A 等 flat / 历史留旧标签）。
> 关联：`docs/STATUS.md` §9（仍延后）、`[[exchange-decoupling-project]]`。

## 1. 背景与问题
- 现有 canonical 符号约定 = ccxt 统一风格 `BASE/USDT:USDT`，但 `USDT` 被**写死**。
- HL 实际用 **USDC** 结算/保证金，适配器把 ccxt 诚实的 `BTC/USDC:USDC` **改写成假的 `BTC/USDT:USDT`** → 日志 / Dashboard / DB 全是误导性的 USDT。
- 语义字段 `quote_currency` 已存在（`CcxtAdapter` 默认 `USDT`、`HyperliquidAdapter` 覆写 `USDC`），但**只用于读余额**（`ccxt_adapter.py` `fetch_balance` 用 `b.get(self.quote_currency)`），**没用于拼符号**。

## 2. 目标 / 非目标
**目标**
- canonical 符号**如实反映结算币**（HL→USDC、OKX→USDT）。
- 计价币**可配置**（适配器类默认 + 可选 config 覆写）。
- **core 零逻辑改动**（符号对 core 仍是不透明 ID）。
- 利用 testnet 数据可弃，**一次切干净**。

**非目标**
- 不做「结构化 instrument 对象」大重构（见 §9 备选 C，列为未来演进项）。
- 不动 OKX 行为（它 USDT 本就诚实）。
- 不强行重贴历史已平网格的标签（留作历史）。

## 3. 关键前提（低风险的根据）
grep 审计确认：
- `to_native` / `to_canonical` **只在各适配器内部被调用**，core / runtime / state / dashboard **零调用**；
- 非测试代码里除适配器自身外**无任何地方解析 / 假设 'USDT'**。

→ **core 把符号当不透明字符串 ID**，换标签是「一致性 + 迁移」问题，**非逻辑问题**。

## 4. 方案设计

### 4.1 canonical 拼接改用 `quote_currency`（核心改动）
| 适配器 | 改动 | 结果 |
|---|---|---|
| `CcxtAdapter` / `OKX` | 拼 canonical 时把写死的 `USDT` 换成 `self.quote_currency` | OKX 仍 `BTC/USDT:USDT`（其 quote=USDT，诚实） |
| `HyperliquidAdapter` | `to_canonical` 用 `self.quote_currency`；`to_native` 同样由 quote 派生 | `BTC/USDC:USDC`（诚实，与 ccxt 原生一致） |

### 4.2 计价币提升为 config（回答「能否 config 设置本币名称」）
- 新增 config 字段 `QUOTE_CURRENCY`（env），**可选**；
- factory 建适配器时：config 有值 → 覆写适配器实例的 `quote_currency`；无值 → 用适配器类默认（HL=USDC、base=USDT）；
- 适配器**同时**用 `self.quote_currency` 拼符号 + 读余额（单一事实源）。

> ⚠️ 设计取舍：结算币本是**交易所固有属性**（HL 恒 USDC），类默认更安全、不易误配；config 覆写主要给「同所多结算」的所（如 Binance USDT-M / USDC-M）。→ 见 §11 开放问题 1。

## 5. 符号流转（改标签必须三处同步）
canonical 在三处必须一致，否则 dedup / restore 失配：
1. **产出**：`list_instruments()`（经 to_canonical）+ **白名单 secret**（静态串）；
2. **存储**：DB `grids.symbol` / `active_symbol`、`grid_orders`、`grid_fills`；
3. **消费**：core / selection / executor 把符号原样传回适配器 `to_native` 调 API。

## 6. 迁移方案（testnet，轻）
**关键风险**：`active_symbol` 唯一约束 + SymbolLock **按 canonical 字符串去重**。若 gt06 存的是 `ETH/USDT:USDT`、而新网格用 `ETH/USDC:USDC` → **同一币会被当两个**，可能 **同时开两个 ETH 网格**！
（注：`to_native` 只取 base，所以旧符号 API 调用其实仍能用——但 dedup 是按整串，会漏。）

**切换必须满足「无混标的活跃币」**，三选一：
- **A（推荐）等 flat 再切**：暂停 scheduler → 等活跃网格自然平 → 改代码 + 白名单 → 部署；
- **B testnet 重置**：清 grids 相关表，干净起步；
- **C 迁移活跃行**：`UPDATE grids SET symbol/active_symbol` 把活跃网格改 USDC（最少停机，但要写迁移脚本）。

历史已平网格：留旧 USDT 标签无害（仅 Dashboard / analytics 里 USDT / USDC 历史分列，可接受或随重置清掉）。

**mainnet（未来）**：直接以 USDC config 起步，**零迁移**。

## 7. 测试方案（TDD，先 RED）
- 适配器单测：HL `to_canonical('BTC/USDC:USDC')=='BTC/USDC:USDC'`、`to_native` 往返、quote 派生；
- OKX 不变：`to_canonical('BTC-USDT-SWAP')=='BTC/USDT:USDT'`；
- config：`QUOTE_CURRENCY` env 解析 → 适配器实例覆写生效；
- `list_instruments()` 对 HL 返回 USDC canonical；
- 余额仍按 `quote_currency` 读；
- 存量断言 `BTC/USDT:USDT`（HL 用例）→ 改 USDC；跑全量双后端绿。

## 9. 备选方案
| 选项 | 评价 |
|---|---|
| A 仅 display 层（日志 / Dashboard 显示 USDC，canonical 不动） | 工作量小，但 DB / config 仍 USDT＝半诚实；已否 |
| **B 本方案（quote_currency 驱动 canonical + config）** | **价值 / 风险最佳，core 零逻辑险，本期做** |
| C 结构化 instrument 对象（base / quote / settle 显式字段，符号只做展示） | 最干净的长远形态，但大重构；**列为未来演进项**，本期不做 |
