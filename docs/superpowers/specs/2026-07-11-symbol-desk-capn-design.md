# SymbolDesk:同币多格操作层币级化(cap=N 系统性收编)设计

> 状态:方案待用户审批。实现前如遇本文未覆盖的分歧点,**不确定就问,勿猜**。
> 前置:spec 2026-07-08-position-ledger-design(v1,已上线 mainnet v29+)。本文是其 v2 演进,
> 非推翻——v1 的不变量、合成成交规范、游标规则、funding 权重全部原样继承。

## 背景与根因

tier2_cap 2→4(2026-07-11 tiercap 扫描,用户定)后同币 3-4 格成为常态。2026-07-11 03:02
mainnet 首个 3 兄弟关格样本触发 `[ledger] WARN close_share ... 2 survivors (cap=2 设计外)`,
巡查审计确认四个问题(审计结论:记账不变量/funding/槽位/选币均已 N-安全,问题集中在**操作层**):

1. **close_share 多幸存格**:残余整笔转给单一 target;多个反号幸存格时取第一个=任意;
   全同号幸存时 `sibs[0]` 兜底造成同号转仓;WARN 常态化刷屏。
2. **PV 币级信号 × N 格同时市价平仓**:PV/趋势类止损是币级信号,同币格必然同触发
   (2026-07-10 SOL/PAXG/NEAR 三对整对退出实证)。cap=4 + MONITOR_PARALLEL=4 最坏情况
   同一薄盘币 4 张市价单近同刻打出(~$6k 名义),自吃滑点。**而其中对冲对(如 PUMP
   gt01/gt07 ±3.2e5)的净敞口≈0,本可以零交易所单出清。**
3. **漂移检测掩护面**:对账口径是 per-symbol Σclaims=交易所净仓,同币格间的归因错误
   (唯一来源=合成转仓行写错)互相抵消后 Σ 检测不到;4 格掩护面积翻倍。
4. **集中度 × 爆仓缓冲交互**:峰值单币敞口 1.67×equity(丝封顶,扫描已计);AL5 缓冲
   核算按分散仓假设,若集中币是 lev3(维持率 16.7%,票池 8 只)尾部缓冲变薄。

**根因**:交易所的资源单位是"币"(HL 每币单一净仓),系统的操作单位是"格"。v1 修了
**归因层**(claims+合成转仓),但**操作层**(关格/止损/丝)仍是逐格直打交易所,格数
上去后互相踩踏与自我冲击按格数放大。系统性解法=把所有"按仓位操作"抬到币级协调,
格级只做决策与记账。

## 设计原则

1. **币级操作,格级记账**:同一币的全部仓位变更(平仓/减仓)经币级协调器一次决策、
   最多一次交易所执行;格间差额全部走 v1 合成转仓(纯记账,mark 价,零费)。
2. **对 N 一般化,对 N=2 退化等价**:所有新逻辑在"1 个幸存格/1 个关格"时逐位等于
   现行为——金标测试零改动通过,升级面可控。
3. **每个移动归因的机制都留下守恒可审计的痕迹**:验证从"抽查"升级为"结构性可证"。

## 组件一:close_share v2——残余比例分摊(问题①)

`position_ledger.py` 单文件改动,纯记账层:

- **反号优先集合**:残余 remaining 的接收方 = 全部反号 claim 幸存格,按 |claim| 比例
  分摊(正是对冲掉本格份额的各方,按其对冲贡献分);
- **无反号幸存格**:按全体幸存格 |claim| 比例分摊(总账守恒,避免单格背全部);
  全部幸存格 claim≈0 时均分(与 funding 兜底同构);
- **N=2 退化**:单一反号幸存格分得 100% ⇒ 与现行为逐位一致;
- **零判定锚统一**:ledger 内全部 `<= ex.min_amount` 改 `EPS = max(ex.min_amount, 1e-9)`
  (收编已批的浮点尘埃一行修,防 prod min_amount=0 时写 1e-14 尘埃转仓行);
- **日志**:WARN 降为 info 单行 `[ledger] closeshare split ...`,列出各接收方与份额;
  真正设计外(无幸存格接收残余)保持现有漂移告警路径不变。

`settle_transfer` 不变(一对合成行);分摊=多次 settle_transfer,天然继承游标/重放/
幂等语义。分摊后本格 claim 归零,重入(CLOSING 续平)残余≈0 不二次转仓——v1 幂等
机制原样适用。

## 组件二:关格集合净额化 + 币级执行序列化(问题②)

### 决策/执行两阶段(cycles.py monitor 轮)

现状:每格 worker 并行地 sync→判止损→**当场** `ex.close(...)`。
改为:

- **阶段 A(并行,只读决策)**:每格 sync + 判 PV/回撤/固定止损,产出关格意向
  `(grid_id, symbol, reason)`,不执行;
- **阶段 B(按币分组,币内串行)**:同币意向合并为一次 `desk.close_set(grid_ids, symbol,
  reason)`;不同币的 close_set 之间可并行(MONITOR_PARALLEL 语义保留在读路径与跨币执行)。

轮换关格(scheduler)与丝触发关格改为调 `close_set`(集合大小 1)——**单格集合退化为
现行路径,全系统只剩一条关格入口**。

### close_set 内部(position_ledger.py 或新 desk 方法)

1. 取币级锁(进程内 per-symbol mutex;跨进程安全继续依赖现有 CLOSING 状态 CAS,
   本设计不新增跨进程锁);
2. 撤全部关格集合的线单与丝(可并行,现有逐格撤单逻辑复用);
3. **内部预净额**:选关格集合中与交易所净仓同号且 |claim| 最大者为**执行格**
   (无同号者=集合净额已被幸存格全额对冲,取 |claim| 最大者,后续 reduce 循环自然
   跳过、残余直接走分摊);其余关格的 claim 全额 settle_transfer 给执行格(mark 价,
   纯记账)——此后其余关格 claim 精确为 0,对冲部分在账本内互相抵消,**永不触碰交易所**;
4. 执行格跑现有 close_share(v23 clamp reduce 循环 ≤3 次 + 组件一的残余比例分摊给
   币上幸存格)——真实成交 1:1 落执行格账本,不破坏"一笔真实成交只归一格"的 DB 约束;
5. 逐格 finalize 记录:各格 pnl 已在步骤 3 按 mark 实现(与今天 close snapshot 按 mark
   实现浮盈亏语义相同);执行格额外承担净额部分的真实滑点(vs mark)。

**收益**:对冲对(±X)出清=零交易所单;4 格同关最多 1 张市价单(净额),总滑点严格 ≤
现状;同币市价单永不并发。
**滑点归因**:落在执行格记录上(有偏但有界;今天逐格自平各吃各的滑点,本质同级)。
若用户要求公平分摊,可后续把执行格真实成交 vs mark 的差额按 |claim| 权重转拨——
默认不做(YAGNI,先保简单)。

## 组件三:结构性验证(问题③)

归因错误的唯一可能来源=合成转仓行(真实成交按 exchange order id→格是确定性映射;
漏摄入会破 Σ 不变量、已有告警)。因此把合成行做成**守恒可审计**:

- **配对守恒**:每次 settle_transfer 的一对合成行带同一 `event_id`(trade_id 已含
  ts+seq,天然可配对;审计断言每 event 恰 2 行、带符号量和=0、同价);
- **reduce 行可交叉核验**:`ledger:reduce` 合成行应有同 client_oid(`%s:close:%d`)的
  真实交易所订单对应;
- **离线审计命令**:`dbadmin verify-ledger`(只读):全量扫 grid_fills 验以上两条 +
  每格账本重放净仓 vs accounting 快照 + per-symbol Σclaims vs 交易所净仓。巡查定期跑,
  masking 从"查不到"变"必留痕"。

无 schema 变更(全部从现有 fills 行派生)。

## 组件四:杠杆感知并发上限(问题④,政策层)

`tier_policy.py` 保持纯层,注入外部数据而非 import 交易所:

- `TierPolicy` 增字段 `lev_caps: tuple = ((3, 1), (5, 2))`——maxlev≤3 的币 cap 1、
  ≤5 的 cap 2、其余走 tier2_cap;
- `cap_for(symbol, tiers, maxlev=None)`:maxlev 由调用方传(scheduler 从 instruments
  已有数据取;dashboard control_compute 同;回测递补用静态当前值近似——历史 maxlev
  不可得,记为已知局限);`maxlev=None` 行为不变(纯层向后兼容);
- 上限口径(1.67×equity 单币峰值)对 lev3 币收敛到 ~0.42×,维持率 16.7% 下缓冲恢复;
- **需扫描验证**:tiercap 扫描口径重跑一版 lev-aware 臂,确认收益代价可接受后再上。

## 测试计划

1. **金标差分**:N=1/N=2 全现有测试零改动通过(组件一退化等价、close_set 单格退化);
2. 比例分摊:3 幸存格(2 反号 1 同号)按 |claim| 比例;全同号;全零均分;尘埃 EPS;
3. 净额化:对冲对同关 → 交易所零下单、双格记录按 mark 实现、Σ 不变量保持;
   4 格混合同关 → 恰一张 reduce 单、量=净额;

6. restore/催化:多转仓行重放复原;跨进程 book catch-up 摄入分摊行(rows=k);
7. lev_caps:纯层单测 + scheduler 注入集成 + `maxlev=None` 向后兼容。
