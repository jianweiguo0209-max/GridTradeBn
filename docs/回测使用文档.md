# 预热与回测 使用文档

> 单一入口：`gridtrade/backtest/backtest_run.py`。
> 一条命令完成「预热（自动下载/复用数据）→ 离线回测 → 汇总 + 明细 CSV」。

---

## 0. TL;DR

```bash
# 推荐：绝对日期窗口（可复现、基线可比）；1m 高保真（选币1h + 持仓1m@Reservoir + funding）
TZ=Asia/Shanghai .venv/bin/python -m gridtrade.backtest.backtest_run 2026-03-01 2026-06-30 1m

# 也支持相对天数（end=今日00:00 UTC）
TZ=Asia/Shanghai .venv/bin/python -m gridtrade.backtest.backtest_run 90 1m
```

- **绝对日期**：`<start> <end> [sim_tf]`，`end` **含当天**（内部 +1 天）。**推荐**——窗口固定，多次/多参数对比才可比。
- **相对天数**：`<days> [sim_tf]`，end=今日 00:00 UTC（当天未过完不纳入）。
- `sim_tf` 默认 `1m`（也可 `1h`）；`1m` 需先配 **AWS 凭证**（Reservoir requester-pays，见 §2）。
- 数据**按天缓存复用**：首跑下载，之后秒开；只增量拉新出现的天。
- 明细 CSV 名含窗口：`data/bt_2026-03-01_2026-06-30_1m_grids.csv`。
- **多核加速**：`BT_WORKERS=8` 并行**选币回放 + 网格仿真**；选币结果按参数+数据指纹自动**磁盘缓存**，同窗口+同参数重跑秒回（`BT_SELECT_CACHE=off` 可关）。见 §3。
- **可回测起点 2025-08-14**：窗口早于 HL API 滚动范围(~200 天)时，1h 选币数据自动改走 Reservoir 归档（1s→1h 重采样，同次下载顺手写全 1m），无需任何开关。

```bash
# 8 核并行 + 选币缓存（首跑算完落盘，重跑秒回）
TZ=Asia/Shanghai BT_WORKERS=8 .venv/bin/python -m gridtrade.backtest.backtest_run 2026-03-01 2026-06-30 1m
```

## 1. 设计总览

三层职责，导入 `backtest_run` **不触发任何网络依赖**（ccxt/adapter/reservoir 全惰性导入），核心可离线测试。

| 函数 | 作用 | 网络 |
|---|---|---|
| `run_backtest(...)` | 纯离线核心：只读 ParquetCache → 选币回放 → 布网 → `simulate_grid_engine` → 聚合 | 无 |
| `prewarm_1h(...)` | phase1：全市场(−黑名单) 1h 选币 OHLCV(HL API，含暖机)；幂等按天复用 | 惰性 |
| `prewarm_sim_and_funding(...)` | phase2：仅选中币 持仓(1m→Reservoir S3 / 其它→HL API) + funding(HL API)；幂等按天复用 | 惰性 |
| `main(argv)` | CLI：全市场解析 → `prewarm_1h` → 选币(一次) → `prewarm_sim_and_funding` → `run_backtest` → summarize → CSV | — |

**为什么持仓要用 1m、且要另找数据源？**
网格成交仿真引擎按「每根 K 线拆 0/15/30/45 秒 4 个 sub-tick」还原触网，只有 **1m** 粒度才有意义；用 1h 会严重低估成交、结果偏悲观。但 Hyperliquid 公共 K 线端点只保留最近 ~5000 根 1m（≈3.5 天），无法回测数月 → 改用 **Reservoir(Hydromancer) S3 归档** 的全历史 1s，重采样成 1m。选币因子仍在 1h 上算（因子在换仓周期粒度上调过，不动），二者由 `sim_timeframe` 解耦。

## 2. 前置：AWS 凭证（仅 `1m` 模式需要）

Reservoir 桶 `s3://hydromancer-reservoir` 是 **requester-pays**（无订阅费，但下载流量记你的 AWS 账户，90 天 8 币约 **1–2 美元**）。

1. IAM 建一个最小权限用户（只读 S3）：
   ```json
   {"Version":"2012-10-17","Statement":[{"Effect":"Allow",
     "Action":["s3:GetObject","s3:ListBucket"],
     "Resource":["arn:aws:s3:::hydromancer-reservoir","arn:aws:s3:::hydromancer-reservoir/*"]}]}
   ```
2. 本机配置（**不要把密钥贴进任何对话/提交**）：
   ```bash
   aws configure          # region 填 ap-northeast-1（桶所在区，跨区多花流量费）
   ```
3. 自检：
   ```bash
   aws sts get-caller-identity
   aws s3 ls s3://hydromancer-reservoir/by_dex/hyperliquid/candles/1s/ --request-payer requester | head
   ```

> `sim_tf=1h` 模式**不需要 AWS**（全部走 HL 公共 API）。

## 3. 命令行用法

```bash
# 绝对日期（推荐）
TZ=Asia/Shanghai .venv/bin/python -m gridtrade.backtest.backtest_run <start> <end> [sim_tf]
# 相对天数
TZ=Asia/Shanghai .venv/bin/python -m gridtrade.backtest.backtest_run <days> [sim_tf]
```

| 参数 | 默认 | 说明 |
|---|---|---|
| `start` `end` | — | 绝对日期 `YYYY-MM-DD`（`end` 含当天）。选币额外向前取 14 天暖机 |
| `days` | `90` | 相对天数模式；end=今日 00:00 UTC |
| `sim_tf` | `1m` | 持仓仿真周期：`1m`（Reservoir，高保真）/ `1h`（HL API，粗略、无需 AWS） |

> 主动止损可配：`strategy_config['active_stop_mode']`（pv/atr/trend/none，默认 pv）
> 及 `strategy_config['pv_config']`（`mult`/`pnl_thr`/`period`/`n`）。见 §6。

> `TZ=Asia/Shanghai` 必须带：选币函数内部读机器时区，须与实盘服务器口径一致。

**票池**：全市场动态（`list_instruments` swap+去重 −黑名单 −逐 run_time PIT `$1M` 成交额地板，从缓存 1h `quote_volume` 前置 24h 重建、无未来函数），与 prod `resolve_live_universe` 同口径；地板/黑名单由 `BT_MIN_QUOTE_VOLUME_24H`/`BT_BLACKLIST`（`backtest_run.py` 顶部常量，env 可覆写）控制。选币数学（55%相对 + 因子排名）不变。

**策略**（在 `backtest_run.py` 顶部常量，镜像 `gridtrade.config`）：
- `HL_STRATEGY`：`DEFAULT_STRATEGY_CONFIG` + `stop_loss_config`（含 `fundingRate_stop_loss=0.0015`）
- `HL_FACTORS`：`Reg_v2_5 / Sgcz_5 / Er_2`

改策略参数：编辑这两个常量；改票池地板/黑名单：设 env `BT_MIN_QUOTE_VOLUME_24H`/`BT_BLACKLIST`；自定义票池/因子：走 §6 的编程接口。

**性能：多核并行 + 选币缓存**（两个 env 旋钮，均可选）

| env | 默认 | 说明 |
|---|---|---|
| `BT_WORKERS` | `1` | `>1` 时**同时并行选币回放（按 run_time 连续切块、多进程）与网格仿真（`ProcessPoolExecutor`）**。选币是长跑瓶颈，CPU-bound 且各 run_time 独立 → 多核近似线性加速（4 个月窗口从数十分钟降到个位数分钟）。结果与串行**逐位一致**（连续切块 + `map` 保序）。 |
| `BT_SELECT_CACHE` | `on` | 选币结果按「选币参数 + 每币缓存天范围指纹」pickle 落盘到 `data/hl_validate/_select_cache/`，**同窗口 + 同参数 + 同数据重跑直接秒回**（跳过整段选币计算）。重新预热改变了缓存天会自动换 key、不返回过期结果。设 `BT_SELECT_CACHE=off` 整体旁路（不读不写）。 |

**1h 数据源自动切换**：`warm_start`（窗口起点 −14 天暖机）早于「今天 −200 天」时，phase1 自动改用
Reservoir 归档重采样 1h（同一 day 文件顺手写全市场 1m，phase2 秒命中）；否则走 HL API（行为与旧版
完全一致）。Reservoir 归档起点 **2025-07-31** → 最早可回测窗口起点 **2025-08-14**；更早会直接报错。
注意：两源同一根 bar 数值可能微差（采集源不同），单次 run 恒单源、无拼缝；同一币缓存可跨 run 混源
（按天分界），缓存天集合变化时选币磁盘缓存指纹会自动换 key。已知盲区：同一窗口在跨过 200 天
边界后重跑（api→reservoir 换源、天集合不变）不会换 key、会复用旧源选币——需要精确重算时
`rm -rf data/hl_validate/_select_cache/` 或 `BT_SELECT_CACHE=off`（两源差异本身仅 bar 级微差）。

**实盘同口径币锁**：`BT_SYMBOL_LOCK=1`（CLI）或 `run_backtest(..., symbol_lock=True)`（编程）
套用与实盘 SymbolLockGate 相同的每币 ≤1 语义——同币在上一格开仓后 12h 锁窗内再被选中即剔除、
不递补（该轮空过）。**默认关**（保历史基线可比）；四窗实测锁会剔除 53-61% 网格、逐格收益和缩水
56-83%——**评估实盘表现请用锁开口径**。近似注记：实盘止损提前退出会提前释放锁（实测提前退出
仅 ~4% 格），本过滤按固定 12h 锁、偏保守。

- 强制重算选币：`rm -rf data/hl_validate/_select_cache/`（或设 `BT_SELECT_CACHE=off` 单次跑）。
- 「就地改写某天旧文件内容」这类指纹盖不住的极少数情况，同样靠删缓存目录 / `off` 兜底。
- 缓存只针对**选币**（1h + 地板 + 黑名单）；持仓/funding 数据仍走 §4 的按天 parquet 缓存。

## 4. 数据源与本地缓存

### 4.1 数据源

| 用途 | 周期 | 来源 | 命名空间 |
|---|---|---|---|
| 选币（因子计算） | 1h | HL 公共 API（近窗口）/ Reservoir S3 1s 重采样（窗口早于 ~200 天，自动切换） | `1h/` |
| 持仓成交仿真 | 1m | Reservoir S3（`sim_tf=1m`）/ HL API（其它） | `1m/` |
| 资金费 PnL + 资金费止损 | — | Hyperliquid 公共 API | `funding/` |

Reservoir 路径：`s3://hydromancer-reservoir/by_dex/hyperliquid/candles/1s/date=YYYY-MM-DD/candles.parquet`（每日一个文件含当天全部 HL 币；region `ap-northeast-1`）。

### 4.2 缓存布局

```
data/hl_validate/{namespace}/{BASE}/{QUOTE:QUOTE}/YYYY-MM-DD.parquet
例：data/hl_validate/1m/BTC/USDC:USDC/2026-06-30.parquet   # 1440 行
data/hl_validate/_select_cache/<hash>.pkl                 # 选币结果缓存（BT_SELECT_CACHE，见 §3）
```

- 复用最小单元 = `(namespace, symbol, day)`，跨窗口/跨 offset 都复用。
- 列定义（`gridtrade/exchanges/base.py`）：
  - OHLCV（`1h`/`1m`）：`symbol, candle_begin_time, open, high, low, close, vol, volCcy, quote_volume`
  - funding：`ts, symbol, fundingRate, realizedRate`
- `data/` 已在 `.gitignore` 中，缓存不入库。

### 4.3 「不完整的天不缓存」语义 ⭐

只缓存**完整**的天；不完整的天不写任何文件（含空哨兵），下次重取：

| 情况 | 行为 |
|---|---|
| 当天(UTC)未过完 | 跳过、不缓存、不触网 → `retry_later` |
| S3/API 拉取 404（未发布）或报错 | 跳过、不写文件 → `retry_later` |
| 日文件已成功下载、但某币当天确无成交 | 该币落**真空哨兵**（0 行），避免反复重取 |

`warm_reservoir_1m` 返回 `{days, rows, skipped_cached, retry_later}`：
- `days`：本次新写入的天数
- `rows`：新写入的 1m 行数
- `skipped_cached`：已缓存被复用的天数（零下载）
- `retry_later`：不完整、留待下次重取的天数

## 5. 输出解读

### 5.1 汇总（`summarize`）

```
n_grids          有效网格数（walk-forward，每小时开一格，持仓 period=12H）
win_rate         单格盈利占比
mean_pnl_ratio   单格收益均值
median_pnl_ratio 单格收益中位
portfolio_return 组合收益 = 各 offset 净值序乘后平均 − 1
offset_equity    每个 offset 的期末净值
exit_reasons     退出原因计数（窗口结束/未触网/固定止损/连续回撤止盈/资金费率止损/pv主动止损/破网/爆仓）
```

### 5.2 明细 CSV

写到 `data/bt_{days}d_{sim_tf}_grids.csv`，每行一个网格：

```
run_time, offset, symbol, entry, grid_num, low, high,
hold_bars, n_fills, pnl_ratio, exit_reason, terminated, funding_missing
```

- `n_fills`：该格触网成交次数（1m 下典型 ~7–11；1h 下常 0–2 → 佐证 1h 偏保守）
- `funding_missing`：该格持仓窗内是否缺 funding 数据（1m 模式应≈0）

### 5.3 参考基线（8 币，v2 布网，最近 3 个月；历史基线——彼时票池写死 8 币，现票池已改全市场动态见 §3）

| 口径 | 有效网格 | 平均成交/格 | 胜率 | 组合收益 | funding |
|---|---|---|---|---|---|
| 1m@Reservoir（真三月） | 2150 | 7.6 | 62.5% | −5.6% | 全计入 |
| 1h（偏保守，低估成交） | 2160 | ~1–2 | 60.2% | −8.1% | 全缺 |

## 6. 编程接口（自定义票池/窗口/参数）

```python
import time, pandas as pd
from gridtrade.backtest.cache import ParquetCache
from gridtrade.backtest.backtest_run import (prewarm_1h, prewarm_sim_and_funding,
                                             run_backtest, summarize, HL_STRATEGY, HL_FACTORS)

cache = ParquetCache('data/hl_validate')
universe = ['BTC/USDC:USDC', 'ETH/USDC:USDC']   # 自定义票池（跳过全市场解析，直接指定）
one_h = 3600_000
end_ms = int(time.time() * 1000)
win_start = end_ms - 30 * 24 * one_h            # 回测窗口
warm_start = end_ms - 44 * 24 * one_h           # 选币暖机（+14 天）

# 1) 预热：phase1 全市场(此处即自定义 universe) 1h → phase2 仅该票池 1m/funding
adapter = prewarm_1h(cache, universe, warm_start, end_ms)
prewarm_sim_and_funding(cache, adapter, universe, win_start, end_ms, sim_timeframe='1m')

# 2) 离线回测（可反复跑、不触网）
df = run_backtest(cache, universe,
                  pd.to_datetime(win_start, unit='ms'), pd.to_datetime(end_ms, unit='ms'),
                  HL_STRATEGY, HL_FACTORS,
                  timeframe='1h', sim_timeframe='1m')
print(summarize(df))
```

`run_backtest` 关键参数：

| 参数 | 默认 | 说明 |
|---|---|---|
| `timeframe` | `'1h'` | 选币因子所用 K 线周期 |
| `sim_timeframe` | `None` | 持仓仿真周期；`None`=沿用 `timeframe`；`'1m'`=解耦高保真 |
| `fee_rate` | `0.00015` | maker 手续费率（网格挂单成交，1.5bps，对齐 HL base 档） |
| `taker_rate` | `0.00045` | taker 手续费率（平仓/止损/破网市价，4.5bps，对齐 HL base 档） |
| `max_rate` | `0.68` | 保证金占用率（每笔挂单量 = cap×lev×max_rate/Σ价）；2026-07-07 由 0.5 对齐实盘 executor 默认 0.68（pnlRatio 振幅 ∝ max_rate，pv_mr_sweep 已证 +0.005 配置饱和区免疫、0.68 下四窗更优） |
| `leverage` | `None` | 覆盖策略杠杆；`None`=用 `strategy_config['leverage']` |
| `min_quote_volume` | `0.0` | 逐 run_time PIT 24h 成交额地板（默认关）；对齐 prod 传 `BT_MIN_QUOTE_VOLUME_24H`（$1M） |
| `blacklist` | `()` | 黑名单符号元组（默认空）；对齐 prod 传 `BT_BLACKLIST` |
| `workers` | `1` | 选币回放 + 网格仿真的并行进程数；`>1` 走多进程，结果与串行逐位一致（CLI 由 `BT_WORKERS` 传入）。也可只并行仿真：`build_grid_tasks`/`simulate_tasks` 各带 `workers` |

> 选币结果磁盘缓存（`BT_SELECT_CACHE`）对编程接口同样透明生效——`run_backtest`/`build_grid_tasks`/`select_grids` 都会命中/落盘（默认开，`off` 旁路）。扫参时选币只算一次：`build_grid_tasks(...)`（一次，含缓存）→ `simulate_tasks(...)`（多次，换不同仿真参数），避免重复选币。

> 只补数据、不回测：直接调 `prewarm_1h(...)` / `prewarm_sim_and_funding(...)`（或 `gridtrade.backtest.reservoir.warm_reservoir_1m` / `gridtrade.backtest.prewarm.*`）。

## 7. 常见问题

**Q: 报 `RuntimeError: Reservoir 未拉到任何 1m 数据`？**
检查：① `aws sts get-caller-identity` 是否成功；② IAM 是否有 `s3:GetObject`；③ region 是否 `ap-northeast-1`；④ 是否窗口太靠前（Reservoir 从 2025-07-31 起）或全落在未发布区间（看 `retry_later`）。

**Q: 最后一两天没数据？**
正常。当天(UTC)未过完不缓存；Reservoir 有 ~1 天发布延迟。过一天再跑会自动补上（`retry_later` 那些天下次重取）。

**Q: 再跑一次会重新下载吗？**
不会。整天全命中即 `skipped_cached` 复用、零下载；只有新出现的天才增量拉。

**Q: 想强制刷新某天？**
删掉该天的 parquet（如 `data/hl_validate/1m/BTC/USDC:USDC/2026-06-30.parquet`），下次预热会重取。

**Q: 不想用 AWS / 只要粗略结果？**
用 `sim_tf=1h`（全走 HL 公共 API，无需 AWS），但注意成交被低估、收益偏保守。

**Q: 想回测 2025 年 8~12 月（HL API 已取不到 1h）？**
直接跑，例如 `TZ=Asia/Shanghai BT_WORKERS=8 .venv/bin/python -m gridtrade.backtest.backtest_run 2025-08-15 2025-10-14 1m`。
窗口早于 API 范围时 1h 自动走 Reservoir（需 AWS 凭证，同 §2）。最早窗口起点 2025-08-14；再早会报错。

## 8. 相关文件

- `gridtrade/backtest/backtest_run.py` — 唯一入口（`main`/`prewarm_1h`/`prewarm_sim_and_funding`/`run_backtest`/`summarize` + HL 常量/BT 票池常量）
- `gridtrade/backtest/reservoir.py` — Reservoir S3 1s→1m/1h 装载器（`warm_reservoir_ohlcv` 多周期一次下载同写）
- `gridtrade/backtest/datasource.py` — HL API 取数层（含「当天不缓存」守卫）
- `gridtrade/backtest/prewarm.py` — `prewarm_ohlcv` / `prewarm_funding` / `resolve_universe`
- `gridtrade/backtest/cache.py` — 按天分片 ParquetCache（+ `list_days` 廉价列举）
- `gridtrade/backtest/selection_replay.py` — 选币回放（point-in-time，多进程并行 `workers`）
- `gridtrade/backtest/select_cache.py` — 选币结果磁盘缓存（key=选币参数+每币缓存天范围指纹，`BT_SELECT_CACHE`）
- `gridtrade/core/grid_engine.py` — 网格成交仿真引擎
- `gridtrade/config.py` — `DEFAULT_STRATEGY_CONFIG` / `DEFAULT_STOP_CFG`
