# 预热与回测 使用文档

> 单一入口：`gridtrade/backtest/backtest_run.py`。
> 一条命令完成「预热（自动下载/复用数据）→ 离线回测 → 汇总 + 明细 CSV」。

---

## 0. TL;DR

```bash
# 推荐：绝对日期窗口（可复现、基线可比）；1m 高保真（选币1h + 持仓1m@Vision + funding）
TZ=Asia/Shanghai .venv/bin/python -m gridtrade.backtest.backtest_run 2026-03-01 2026-06-30 1m

# 也支持相对天数（end=今日00:00 UTC）
TZ=Asia/Shanghai .venv/bin/python -m gridtrade.backtest.backtest_run 90 1m
```

- **绝对日期**：`<start> <end> [sim_tf]`，`end` **含当天**（内部 +1 天）。**推荐**——窗口固定，多次/多参数对比才可比。
- **相对天数**：`<days> [sim_tf]`，end=今日 00:00 UTC（当天未过完不纳入）。
- `sim_tf` 默认 `1m`（也可 `1h`）；数据源为币安官方归档 `data.binance.vision`，**免费、无需鉴权**，没有额外前置配置（见 §2）。
- 数据**按天缓存复用**：首跑下载，之后秒开；只增量拉新出现的天。
- 明细 CSV 名含窗口：`data/bt_2026-03-01_2026-06-30_1m_grids.csv`。
- **多核加速**：`BT_WORKERS=8` 并行**选币回放 + 网格仿真**；选币结果按参数+数据指纹自动**磁盘缓存**，同窗口+同参数重跑秒回（`BT_SELECT_CACHE=off` 可关）。见 §3。
- **全历史可回测**：Vision 归档覆盖 2019-09（币安 U 本位合约上线）至今，含已退市合约（无幸存者偏差）；窗口越早/合约越多，首跑下载量越大，建议配 `BT_WORKERS` 并行（数据体量见 §2、全历史示例见 §7）。

```bash
# 8 核并行 + 选币缓存（首跑算完落盘，重跑秒回）
TZ=Asia/Shanghai BT_WORKERS=8 .venv/bin/python -m gridtrade.backtest.backtest_run 2026-03-01 2026-06-30 1m
```

## 1. 设计总览

三层职责，导入 `backtest_run` **不触发任何网络依赖**（ccxt/adapter/vision 全惰性导入），核心可离线测试。

| 函数 | 作用 | 网络 |
|---|---|---|
| `run_backtest(...)` | 纯离线核心：只读 ParquetCache → 选币回放 → 布网 → `simulate_grid_engine` → 聚合 | 无 |
| `prewarm_1h(...)` | phase1：全市场(−黑名单) 1h 选币 OHLCV(Vision 归档批量 + API 尾补，含暖机)；幂等按天复用 | 惰性 |
| `prewarm_sim_and_funding(...)` | phase2：仅选中币 持仓(Vision 归档 + API 尾补) + funding(Vision 归档 + API 尾补)；幂等按天复用 | 惰性 |
| `main(argv)` | CLI：全市场解析 → `prewarm_1h` → 选币(一次) → `prewarm_sim_and_funding` → `run_backtest` → summarize → CSV | — |

**为什么持仓要用 1m、数据从哪来？**
网格成交仿真引擎按「每根 K 线拆 0/15/30/45 秒 4 个 sub-tick」还原触网，只有 **1m** 粒度才有意义；用 1h 会严重低估成交、结果偏悲观。币安公共 K 线端点本身不限历史深度，但全历史、全市场（含已退市合约）逐页拉取 1m 极慢且耗费率限额 → 改用官方批量归档 **`data.binance.vision`**（月度/日度 zip，`.CHECKSUM` 校验），一次性批量拉回全历史，速度和完整性都远优于逐根分页。选币因子仍在 1h 上算（因子在换仓周期粒度上调过，不动），二者由 `sim_timeframe` 解耦。归档相对当下有 1-2 天发布滞后，窗口末尾未覆盖的天由公共 API 自动尾补（同一份按天缓存，来源对下游透明）。

## 2. 数据源：币安官方归档（Vision）

`data.binance.vision` 是币安官方公开归档，**免费、无需注册/API Key/任何鉴权**，公网可直接 HTTPS 访问，无需像旧数据源那样单独配置云凭证。

- **覆盖范围**：U 本位永续合约（USDT-M），2019-09（`fapi` 上线）至今；**含已退市合约**——目录列举取的是归档全量历史合约，天然无幸存者偏差（§3 票池）。
- **归档结构**：
  - 月度 K 线（为主）：`.../data/futures/um/monthly/klines/{SYMBOL}/{1m,1h}/{SYMBOL}-{tf}-{YYYY-MM}.zip`
  - 日度 K 线（当月尾部回退用）：`.../data/futures/um/daily/klines/{SYMBOL}/{1m,1h}/{SYMBOL}-{tf}-{YYYY-MM-DD}.zip`
  - 月度资金费：`.../data/futures/um/monthly/fundingRate/{SYMBOL}/{SYMBOL}-fundingRate-{YYYY-MM}.zip`
  - 每个 zip 配 `.CHECKSUM`（sha256）；下载后校验，不一致按「未发布」处理——不落哨兵，下次重取，不会写入损坏数据。
- **尾部补齐**：归档发布滞后 1-2 天，窗口末尾这几天由公共 API（`ccxt.binanceusdm`，只读行情/资金费，无需 API Key）自动尾补；两条路径写入同一份按天缓存，`run_backtest` 对数据来源零感知。
- **无需任何前置步骤**——不需要 IAM、`aws configure`、`aws sts get-caller-identity` 之类的操作，只要能访问公网即可。
- **缓存根目录**：默认 `<repo>/data/binance`（`gridtrade.backtest.vision.default_cache_root()`），可用 env `BT_DATA_DIR` 覆写（挂到独立磁盘/共享缓存卷时用），详见 §4。
- **数据体量**：全历史 1h 全市场（约数百合约含退市）≈ 数 GB；1m 仅按窗口拉选中币，全历史全币 1m 上限数十 GB——分层预热（phase1 全市场 1h、phase2 仅选中币 1m/funding）天然控制实际落盘量。磁盘不足时可按窗口分段回测（`<start> <end>` 参数已支持）。

## 3. 命令行用法

```bash
# 绝对日期（推荐）
TZ=Asia/Shanghai .venv/bin/python -m gridtrade.backtest.backtest_run <start> <end> [sim_tf]
# 相对天数
TZ=Asia/Shanghai .venv/bin/python -m gridtrade.backtest.backtest_run <days> [sim_tf]
```

| 参数 | 默认 | 说明 |
|---|---|---|
| `start` `end` | — | 绝对日期 `YYYY-MM-DD`（`end` 含当天）。选币额外向前取 14 天暖机 |
| `days` | `90` | 相对天数模式；end=今日 00:00 UTC |
| `sim_tf` | `1m` | 持仓仿真周期：`1m`（Vision 归档+API 尾补，高保真，需下载选中币的分钟线）/ `1h`（复用 phase1 已预热的全市场 1h 数据，零额外下载，但成交仿真更粗略、结果偏保守） |

> 主动止损可配：`strategy_config['active_stop_mode']`（pv/atr/trend/none，默认 pv）
> 及 `strategy_config['pv_config']`（`mult`/`pnl_thr`/`period`/`n`）。见 §6。

> `TZ=Asia/Shanghai` 必须带：选币函数内部读机器时区，须与实盘服务器口径一致。

**独立预热（只拉数据、不跑回测）**：`backtest_run` 的 CLI 会按窗口自动预热，一般不需要手工调用；
但单独补数据、修复缺口、或给多个窗口共享预热时可以直接用 `vision_sync`：

```bash
# 预热全历史归档：1h + 1m + funding 三种命名空间，全市场（含退市）
.venv/bin/python -m gridtrade.backtest.vision_sync 2019-09-01 2026-07-13 --tf 1h,1m,funding

# 只拉指定币种、指定周期、指定并行度
.venv/bin/python -m gridtrade.backtest.vision_sync 2026-01-01 2026-06-30 \
    --tf 1m --symbols BTC/USDT:USDT,ETH/USDT:USDT --workers 8
```

- `--tf`：逗号分隔 `1m,1h,funding`（默认 `1m`）。
- `--symbols`：留空=归档目录全量合约（含退市）；也可显式指定 canonical 逗号分隔列表。
- `--workers`：线程池并行度，覆盖 env `BT_VISION_WORKERS`（默认 `8`）。
- 幂等、可断点续跑：整月/整天已缓存直接跳过，中断后重跑只补缺口。

**票池**：全市场动态——候选源为 Vision 归档目录的全量历史合约（`vision.list_archive_symbols()`，含已退市，无幸存者偏差）−黑名单 −逐 run_time PIT 24h 成交额**前 55% 相对口径**（`BT_UNIVERSE_TOP_PCT` 默认 0.55，对齐 prod `UNIVERSE_TOP_VOLUME_PCT`；24h 量从缓存 1h `quote_volume` 前置 24 根重建、无未来函数；spec 2026-07-14-universe-top-volume-pct）；绝对地板 `BT_MIN_QUOTE_VOLUME_24H`（默认 0=停用，机制保留可叠加）与黑名单 `BT_BLACKLIST` 同为 `backtest_run.py` 顶部常量、env 可覆写。选币层为纯三因子排名——legacy 的「交易额分位占比 ≤0.55」复合过滤已取消（2026-07-14 用户定，票池层相对口径接管）。**候选源有意不同**——prod 用 `list_instruments` 只看当前活跃合约，回测用归档全量（含退市）以避免历史评估的幸存者偏差。

**策略**（在 `backtest_run.py` 顶部常量，镜像 `gridtrade.config`）：
- `BT_STRATEGY`：`DEFAULT_STRATEGY_CONFIG` + `stop_loss_config`（含 `fundingRate_stop_loss=0.0015`）
- `BT_FACTORS`：`Reg_v2_5 / Sgcz_5 / Er_2`

改策略参数：编辑这两个常量；改票池地板/黑名单：设 env `BT_MIN_QUOTE_VOLUME_24H`/`BT_BLACKLIST`；自定义票池/因子：走 §6 的编程接口。

**性能：多核并行 + 选币缓存**（两个 env 旋钮，均可选）

| env | 默认 | 说明 |
|---|---|---|
| `BT_WORKERS` | `1` | `>1` 时**同时并行选币回放（按 run_time 连续切块、多进程）与网格仿真（`ProcessPoolExecutor`）**。选币是长跑瓶颈，CPU-bound 且各 run_time 独立 → 多核近似线性加速（4 个月窗口从数十分钟降到个位数分钟）。结果与串行**逐位一致**（连续切块 + `map` 保序）。 |
| `BT_SELECT_CACHE` | `on` | 选币结果按「选币参数 + 每币缓存天范围指纹」pickle 落盘到 `data/binance/_select_cache/`，**同窗口 + 同参数 + 同数据重跑直接秒回**（跳过整段选币计算）。重新预热改变了缓存天会自动换 key、不返回过期结果。设 `BT_SELECT_CACHE=off` 整体旁路（不读不写）。 |

**数据源分层，对窗口新旧一视同仁**：无论窗口远近，1h（选币）与 1m/funding（仿真）都优先从 Vision 归档批量拉取；归档发布滞后 1-2 天，窗口末尾这部分天由公共 API 自动尾补——两条路径写入同一份按天缓存，下游对来源零感知。Vision 归档起点为 2019-09（币安 U 本位合约上线）；再往前没有归档数据，对应天返回空（不是报错），可回测的实际起点约等于 2019-09。已退市合约同样保留在归档目录中，全市场票池解析（`list_archive_symbols`）因此天然覆盖它们。

注意：若某天先由 API 尾补写入缓存（当时归档还未发布），之后 Vision 官方归档才补发布，缓存**不会**自动用归档覆盖 API 版本（按天存在即跳过、不重取）——两者同源币安数据，理论上应一致；如需要精确统一用归档口径，删对应天的 parquet 重新预热即可。

- 强制重算选币：`rm -rf data/binance/_select_cache/`（或设 `BT_SELECT_CACHE=off` 单次跑）。
- 「就地改写某天旧文件内容」这类指纹盖不住的极少数情况，同样靠删缓存目录 / `off` 兜底。
- 缓存只针对**选币**（1h + 地板 + 黑名单）；持仓/funding 数据仍走 §4 的按天 parquet 缓存。

**实盘同口径急动刹车（MarketShockBrake）**：`run_backtest(..., shock_brake=(4, 0.04, 2))`
——|票池中位数 4h 收益|≥4% 的 rt 起暂停开格 2h（blocked rt 候选整轮剔除、不递补,只影响开格）。
信号数学与实盘 `runtime/shock.py` 同源（守卫测试 `tests/backtest/test_shock_replay.py` 逐 rt 钉一致）。
**默认 None=关**（保历史基线可比）；**对齐实盘现配置（testnet v78+）请传 `(4, 0.04, 2)`**。
回测依据/参数扫描见 memory `shock-brake-verdict` 与 `data/tiercmp/sb_results.csv`。

**实盘同口径币锁**：`BT_SYMBOL_LOCK=1`（CLI）或 `run_backtest(..., symbol_lock=True)`（编程）
套用与实盘 SymbolLockGate 相同的每币 ≤1 语义——同币在上一格开仓后 12h 锁窗内再被选中即剔除、
不递补（该轮空过）。**默认关**（保历史基线可比）；四窗实测锁会剔除 53-61% 网格、逐格收益和缩水
56-83%——**评估实盘表现请用锁开口径**。近似注记：实盘止损提前退出会提前释放锁（实测提前退出
仅 ~4% 格），本过滤按固定 12h 锁、偏保守。

## 4. 数据源与本地缓存

### 4.1 数据源

| 用途 | 周期 | 来源 | 命名空间 |
|---|---|---|---|
| 选币（因子计算） | 1h | Vision 归档批量 + 币安公共 API 尾补（归档滞后 1-2 天） | `1h/` |
| 持仓成交仿真 | 1m | Vision 归档批量 + 币安公共 API 尾补（`sim_tf=1m` 时） | `1m/` |
| 资金费 PnL + 资金费止损 | — | Vision 归档批量 + 币安公共 API 尾补 | `funding/` |

Vision 归档路径：K 线 `https://data.binance.vision/data/futures/um/{monthly,daily}/klines/{SYMBOL}/{tf}/{SYMBOL}-{tf}-{YYYY-MM[-DD]}.zip`；资金费 `.../fundingRate/{SYMBOL}/{SYMBOL}-fundingRate-{YYYY-MM}.zip`；月度为主、当月尾部用日度回退；每个 zip 配 `.CHECKSUM`（sha256）。

### 4.2 缓存布局

```
data/binance/{namespace}/{BASE}/{QUOTE:QUOTE}/YYYY-MM-DD.parquet
例：data/binance/1m/BTC/USDT:USDT/2026-06-30.parquet     # 1440 行
data/binance/_select_cache/<hash>.pkl                    # 选币结果缓存（BT_SELECT_CACHE，见 §3）
```

- 根目录默认 `<repo>/data/binance`，可用 env `BT_DATA_DIR` 覆写，例：
  `BT_DATA_DIR=/mnt/btcache TZ=Asia/Shanghai .venv/bin/python -m gridtrade.backtest.backtest_run 90 1m`
- 复用最小单元 = `(namespace, symbol, day)`，跨窗口/跨 offset 都复用。
- 列定义（`gridtrade/exchanges/base.py`）：
  - OHLCV（`1h`/`1m`）：`symbol, candle_begin_time, open, high, low, close, vol, volCcy, quote_volume`
  - funding：`ts, symbol, fundingRate, realizedRate`
- `data/` 已在 `.gitignore` 中，缓存不入库。

### 4.3 「不完整的天不缓存」语义 ⭐

只缓存**完整**的天；不完整的天不写任何文件（含空哨兵），下次重取：

| 情况 | 行为 |
|---|---|
| 当天(UTC)未过完 | 跳过、不缓存、不触网 → `retry_later` |
| 归档/API 拉取 404（未发布）或报错、CHECKSUM 校验不通过 | 跳过、不写文件 → `retry_later` |
| 归档文件已成功下载、但某币当天确无成交（或该合约当月未上市） | 该币落**真空哨兵**（0 行），避免反复重取 |

`warm_vision` 返回 `{<namespace>: {rows, files}, skipped_cached, retry_later, empty_days}`（`vision_sync`/`prewarm_1h`/`prewarm_sim_and_funding` 内部打印的 stats 即此结构）：
- `<namespace>.rows`/`.files`：本次新写入的行数 / 新下载的归档文件数（monthly 或 daily zip）
- `skipped_cached`：已缓存（该月/该天全命中）被复用，零下载
- `retry_later`：未发布或请求失败，留待下次重取
- `empty_days`：确认无数据（真空哨兵），避免反复重试

## 5. 输出解读

### 5.1 汇总（`summarize`）

```
n_grids          有效网格数（walk-forward，每小时开一格，持仓 period=12H）
win_rate         单格盈利占比
mean_pnl_ratio   单格收益均值
median_pnl_ratio 单格收益中位
portfolio_return 组合收益 = 各 offset 净值序乘后平均 − 1
offset_equity    每个 offset 的期末净值
exit_reasons     退出原因计数（窗口结束/未触网/固定止损/连续回撤止盈/资金费率止损/pv主动止损/破网/爆仓）
```

### 5.2 明细 CSV

写到 `data/bt_{days}d_{sim_tf}_grids.csv`，每行一个网格：

```
run_time, offset, symbol, entry, grid_num, low, high,
hold_bars, n_fills, pnl_ratio, exit_reason, terminated, funding_missing
```

- `n_fills`：该格触网成交次数（1m 下典型 ~7–11；1h 下常 0–2 → 佐证 1h 偏保守）
- `funding_missing`：该格持仓窗内是否缺 funding 数据（1m 模式应≈0）

## 6. 编程接口（自定义票池/窗口/参数）

```python
import time, pandas as pd
from gridtrade.backtest.cache import ParquetCache
from gridtrade.backtest.backtest_run import (prewarm_1h, prewarm_sim_and_funding,
                                             run_backtest, summarize, BT_STRATEGY, BT_FACTORS)

cache = ParquetCache('data/binance')
universe = ['BTC/USDT:USDT', 'ETH/USDT:USDT']   # 自定义票池（跳过全市场解析，直接指定）
one_h = 3600_000
end_ms = int(time.time() * 1000)
win_start = end_ms - 30 * 24 * one_h            # 回测窗口
warm_start = end_ms - 44 * 24 * one_h           # 选币暖机（+14 天）

# 1) 预热：phase1 全市场(此处即自定义 universe) 1h → phase2 仅该票池 1m/funding
adapter = prewarm_1h(cache, universe, warm_start, end_ms)
prewarm_sim_and_funding(cache, adapter, universe, win_start, end_ms, sim_timeframe='1m')

# 2) 离线回测（可反复跑、不触网）
df = run_backtest(cache, universe,
                  pd.to_datetime(win_start, unit='ms'), pd.to_datetime(end_ms, unit='ms'),
                  BT_STRATEGY, BT_FACTORS,
                  timeframe='1h', sim_timeframe='1m')
print(summarize(df))
```

`run_backtest` 关键参数：

| 参数 | 默认 | 说明 |
|---|---|---|
| `timeframe` | `'1h'` | 选币因子所用 K 线周期 |
| `sim_timeframe` | `None` | 持仓仿真周期；`None`=沿用 `timeframe`；`'1m'`=解耦高保真 |
| `fee_rate` | `0.0002` | maker 手续费率（网格挂单成交，2bps，对齐币安 USDT-M VIP0 无折扣档） |
| `taker_rate` | `0.0005` | taker 手续费率（平仓/止损/破网市价，5bps，对齐币安 USDT-M VIP0 无折扣档） |
| `max_rate` | `0.68` | 保证金占用率（每笔挂单量 = cap×lev×max_rate/Σ价）；2026-07-07 由 0.5 对齐实盘 executor 默认 0.68（pnlRatio 振幅 ∝ max_rate，pv_mr_sweep 已证 +0.005 配置饱和区免疫、0.68 下四窗更优） |
| `leverage` | `None` | 覆盖策略杠杆；`None`=用 `strategy_config['leverage']` |
| `min_quote_volume` | `0.0` | 逐 run_time PIT 24h 成交额**绝对地板**（默认关；prod 已改相对口径，此项仅作可叠加的额外地板） |
| `top_volume_pct` | `0.0` | 逐 run_time PIT 24h 成交额**前 pct 相对口径**（对齐 prod 传 `0.55`；CLI 默认 `BT_UNIVERSE_TOP_PCT=0.55`） |
| `blacklist` | `()` | 黑名单符号元组（默认空）；对齐 prod 传 `BT_BLACKLIST` |
| `workers` | `1` | 选币回放 + 网格仿真的并行进程数；`>1` 走多进程，结果与串行逐位一致（CLI 由 `BT_WORKERS` 传入）。也可只并行仿真：`build_grid_tasks`/`simulate_tasks` 各带 `workers` |

> 选币结果磁盘缓存（`BT_SELECT_CACHE`）对编程接口同样透明生效——`run_backtest`/`build_grid_tasks`/`select_grids` 都会命中/落盘（默认开，`off` 旁路）。扫参时选币只算一次：`build_grid_tasks(...)`（一次，含缓存）→ `simulate_tasks(...)`（多次，换不同仿真参数），避免重复选币。

> 只补数据、不回测：直接调 `prewarm_1h(...)` / `prewarm_sim_and_funding(...)`（或 `gridtrade.backtest.vision.warm_vision` / `gridtrade.backtest.prewarm.*`）。

## 7. 常见问题

**Q: 报 `RuntimeError: <tf> 数据完全缺失`？**
检查：① 网络能否直连 `data.binance.vision`（归档下载）与 `s3-ap-northeast-1.amazonaws.com`（Vision 目录列举用该域名）；② 币种/窗口是否确实无归档（窗口早于该合约上市月、或早于全站起点 2019-09）；③ 看返回 stats 里的 `retry_later`——非 0 说明有天暂时未发布或请求失败，过一段时间重跑通常会自动补上。

**Q: 最后一两天没数据？**
正常。当天(UTC)未过完不缓存；Vision 月度归档对当月有 1-2 天发布滞后，但同一天会被公共 API 自动尾补，一般感知不到。真出现 gap 时，`retry_later` 记录的天下次重跑会自动补上。

**Q: 再跑一次会重新下载吗？**
不会。整天/整月全命中即 `skipped_cached` 复用、零下载；只有新出现的天才增量拉。

**Q: 想强制刷新某天？**
删掉该天的 parquet（如 `data/binance/1m/BTC/USDT:USDT/2026-06-30.parquet`），下次预热会重取。若某天先由 API 尾补写入、之后归档才补发布，缓存不会自动用归档覆盖（按天存在即跳过）——两者同源币安数据，通常一致；确需统一口径同样删该天重取。

**Q: 需要配置 AWS/IAM 之类的凭证吗？**
不需要。`data.binance.vision` 是币安官方公开归档，免费、无鉴权，公网可直接 HTTPS 访问（详见 §2）。

**Q: 不想拉全历史 / 只要粗略结果？**
用 `sim_tf=1h`（选币和仿真都复用 1h 数据，跳过 1m 归档下载），但注意成交被低估、收益偏保守；或缩小 `<start> <end>` 窗口只覆盖关心的区间。

**Q: 想跑全历史回测（2019 年至今）？**
直接跑，例如：

```bash
TZ=Asia/Shanghai BT_WORKERS=8 .venv/bin/python -m gridtrade.backtest.backtest_run 2020-01-01 2026-06-30 1m
```

首跑要下载/解析全市场（含退市）多年归档，耗时随窗口跨度和合约数增长（数据体量见 §2）；建议先配 `BT_WORKERS` 并行，必要时用 `vision_sync`（见 §3）提前分批预热。归档起点 2019-09（币安 U 本位合约上线）；更早没有数据（对应天返回空，不是报错）。
